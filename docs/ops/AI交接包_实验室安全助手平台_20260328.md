# AI 交接包：实验室安全助手平台（2026-03-28）

## 1. 项目目标（最终形态）

本项目目标是从“问答工具”升级为“实验室安全助手平台”，核心能力如下：

1. 面向常规实验室安全问题，提供基于知识库的可追溯回答。
2. 对高风险问题触发规则约束（拒答、重定向、应急优先）。
3. 形成可持续的数据处理流水线：采集 -> 清洗 -> 入库 -> 评测 -> 发布门禁。
4. 尽量 AI 全自动，人工仅做后期抽检与验收复核。

## 2. 当前执行原则（必须遵守）

1. 数据必须标注来源，可追溯到原始链接或原始文件。
2. 现阶段优先云端 API 验证全链路，后续再切换到本地模型。
3. 默认尽量非人工操作，人工流程只保留“最终复核”。
4. 不使用 Apifox；外部调用以当前仓库已有脚本与 GitHub Actions 为主。

## 3. 当前阶段进展（已完成）

### 3.1 数据与知识库

1. 已有主知识库文件：`knowledge_base_curated.csv`。

## 7. 下一步执行计划（建议直接按顺序做）

### Step 1：数据侧补强（高优先级）

1. 扩充高质量、可访问、可追溯的公开来源（优先官方/高校/规范类 PDF）。
2. 按统一字段写入 `data_sources` 清单，并自动预抓取状态化。
3. 对 blocked/error 链接自动补链，补链失败进入人工补录目录。

完成标准：

1. 新增一批可用来源并产出可追溯清单。
2. 所有新增条目有 `source_url` 与 `source_title`。

### Step 2：评测集 V2 扩容（高优先级）

1. 从 `eval_set_v1.csv` 扩展到更大覆盖（常规/应急/拒答/模糊提问）。
2. 同步更新验收指标与失败分簇分析。

完成标准：

1. 每周自动回归能稳定跑。
2. `weekly_gate_ops` 趋势可读，能指导修复优先级。

### Step 3：平台体验与演示（中优先级）

1. 把 `web_demo` 和问答链路做稳定接入（团队可在线演示）。
2. 输出可答辩的一键演示脚本与最小可复现步骤。

完成标准：

1. 新队员按文档可 30 分钟内复现演示。

### Step 4：本地模型迁移准备（中优先级）

1. 先做云 API vs 本地模型 A/B（同一评测集）。
2. 通过阈值后再讨论切流，不直接替换线上链路。

完成标准：

1. 迁移决策有数据支撑（非主观判断）。

## 8. 交接约定（建议下一个 AI 遵守）

1. 每次改动都给出“Task 文本 + 直接执行 + 结果回报”。
2. 每次重要修改后提交并推送 GitHub。
3. 禁止破坏性命令（例如 reset --hard）。
4. 如果发现上下文与仓库不一致，先 pull 后再改。

## 9. 风险与注意事项

1. 部分历史文档存在编码混杂现象（内容可读性不一致），后续建议统一 UTF-8 重整。
2. 线上 API 可用性会直接影响门禁结果，需要看 route-health 指标，不要只盯 QA 分数。
3. 自动化程度高时，更要保证来源追溯与审计日志完整。
