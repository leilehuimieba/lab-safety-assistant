# 实验室安全小助手演示脚本

## 文档目的

本脚本用于项目答辩、阶段汇报和现场演示，帮助团队在 5 至 8 分钟内完整展示“实验室安全小助手”的项目背景、系统能力、知识库基础、回答效果和后续优化方向。

## 演示目标

- 让评审快速理解项目要解决的问题
- 展示系统已经具备的核心能力，而不是只讲设想
- 用典型问题验证知识库问答、规则约束和应急场景处理能力
- 说明当前系统边界和下一步优化路径

## 演示环境建议

- 演示设备：一台联网电脑
- 演示方式：本地自部署 Dify 原型
- 准备页面：
  - 应用对话页
  - 知识库或工作流页面
  - 本仓库 GitHub 首页
- 演示前确认：
  - Dify 服务可访问
  - 知识库已导入并可检索
  - 工作流回答节点已绑定 LLM 实际输出
  - 至少完成 5 个典型问题的联调测试

## 角色分工建议

- 主讲人：负责介绍项目背景、目标、创新点和演示收尾
- 操作人：负责现场输入问题、切换页面、展示知识库和评测材料
- 记录人：负责记录现场问题与评审反馈，便于后续优化

## 推荐演示时长

- 项目引入：1 分钟
- 系统展示：1 分钟
- 现场问答演示：3 至 4 分钟
- 总结与展望：1 至 2 分钟

## 开场讲解稿

### 1. 项目背景

可以这样讲：

“高校实验室中常见的安全问题不是完全没有制度，而是学生在实际操作前、操作中和应急情况下，很难快速得到结构化、可执行、场景化的安全建议。我们希望构建一个实验室安全小助手，帮助用户围绕实验室制度、危险操作、应急处置和常见合规问题获得辅助回答。”

### 2. 项目目标

可以这样讲：

“我们当前阶段的目标不是做一个泛化聊天机器人，而是做一个面向实验室安全场景的原型系统。它有三个核心特点：一是基于知识库问答，二是对高风险问题进行规则约束，三是能够通过评测集和召回测试持续优化。”

### 3. 技术路线

可以这样讲：

“现阶段我们采用的是 Dify + RAG + 规则库 的路线。知识经过清洗后进入知识库，用户提问经过检索召回，再由大模型生成回答，同时通过规则库对危险场景进行约束。系统还配套了评测集和验收指标，用于后续持续优化。”

## 现场问答演示脚本

下面的问题建议按“常规问答 -> 危险行为 -> 应急处置”的顺序演示。

### 演示问题 1：通风柜什么时候必须开启？

- 目标：展示常规知识库问答能力
- 预期要点：
  - 挥发性、刺激性、有毒试剂操作时必须开启
  - 应保持前窗在安全刻度范围
  - 应确认风速和报警状态正常
- 讲解重点：
  - 说明系统可以回答日常规范类问题
  - 强调回答不是泛泛而谈，而是有操作要点

### 演示问题 2：乙醇应如何储存？

- 目标：展示危化品储存问答能力
- 预期要点：
  - 存放于防火安全柜
  - 远离热源和明火
  - 容器密闭并标签清晰
- 讲解重点：
  - 说明系统可以覆盖危化品基本管理要求
  - 强调该类知识适合新生和新成员学习

### 演示问题 3：盐酸和漂白水能混吗？

- 目标：展示规则约束与危险行为识别能力
- 预期要点：
  - 明确禁止混合
  - 提示有毒气体风险
  - 如已混合，应停止操作并按应急预案处理
- 讲解重点：
  - 这类问题不能只给知识解释，还要给安全边界
  - 展示系统的“纠偏”能力

### 演示问题 4：有人触电了怎么办？

- 目标：展示电气应急场景能力
- 预期要点：
  - 先断电
  - 禁止直接接触触电者
  - 使用绝缘物隔离
  - 呼叫急救并按预案处置
- 讲解重点：
  - 说明系统能处理应急类问题
  - 强调应急回答以步骤和优先级为主

### 演示问题 5：实验室着火怎么办？

- 目标：展示通用消防应急能力
- 预期要点：
  - 立即停止实验
  - 切断电源和气源
  - 小火在安全前提下使用灭火器
  - 无法控制时立即撤离并报警
- 讲解重点：
  - 展示系统对通用高风险场景的支持
  - 可顺势引出“消防、危化品、电气”三类知识覆盖

## 演示中可主动强调的亮点

- 当前原型不是只做对话，而是有知识库、规则库、评测集和调参记录
- 对明显危险问题，系统会给出安全约束而不是鼓励操作
- 已经完成首轮知识库清洗和召回调参，说明项目有工程推进基础
- 项目后续可以平滑接入 Embedding、本地模型和更正式的数据源

## 如果现场回答不理想，如何解释

### 情况 1：回答较笼统

建议说法：

“当前版本已经能覆盖核心场景，但正式数据源还没有完全补齐，因此下一阶段会重点引入学校制度文件、SOP 和 MSDS，进一步提高回答的细致程度和可追溯性。”

### 情况 2：召回有噪声

建议说法：

“我们已经完成第一轮召回测试，当前主要使用倒排检索。下一步会接入 Embedding、混合检索和重排序模型，降低噪声，提高正确条目排序。”

### 情况 3：个别问题未命中

建议说法：

“这也说明实验室安全系统必须依赖正式数据源建设。当前仓库里已经有知识库模板、规则库和评测集，下一步重点就是扩展正式来源并迭代知识条目。”

## 演示收尾讲解稿

可以这样讲：

“总体来说，我们已经完成了实验室安全小助手的 MVP 原型：具备结构化知识库、规则库、评测集和初步调参能力。下一阶段，我们会继续补充正式安全资料来源，引入更高质量检索和本地模型部署，使其从一个可展示原型逐步发展成一个可落地的实验室安全辅助系统。”
