# 实验室安全助手平台 3 分钟演示脚本

## 1. 演示目标

这次演示不是展示一个普通聊天机器人，而是展示一个面向实验室安全场景的闭环平台。3 分钟内要让评委看到 4 件事：

1. 它能回答常见实验室安全问题，而且回答有依据。
2. 它能在高风险场景下阻断不合规开工。
3. 它能把日常运行情况沉淀到管理端看板。
4. 它能对事故进行复盘并形成整改闭环。

## 2. 演示前准备

- 演示地址：`http://175.178.90.193:8088/`
- 预先打开首页，确认页面可访问。
- 最新统一口径：
  - 线上知识库正式导入：`398` 条成功
  - 最新 20 题正式回归：`20/20`
  - 连续 3 轮真实回归：`3/3 PASS`
  - 当前性能状态：正确率稳定达标，延迟已优化但仍有波动
- 预先准备 4 个模块的入口：
  - 问答
  - 开工前检查清单
  - 管理端看板
  - 事故复盘
- 若现场网络波动，优先使用下面脚本中的固定问题，不临时发挥。

## 3. 3 分钟演示链路

### 0:00 - 0:20 开场定位

建议话术：

“我们做的不是一个泛化聊天工具，而是一个实验室安全助手平台。它围绕实验室日常使用中的 4 个高频环节来设计：事前问答、开工前检查、过程管理看板、事后事故复盘，目标是把实验室安全从‘查制度’变成‘可执行、可追踪、可闭环’。”

### 0:20 - 1:00 问答模块

操作：

1. 进入问答区。
2. 输入问题：`危险实验开始前最关键的阻断项是什么？`

建议话术：

“这里先展示最基础但最关键的一层，也就是知识问答。系统会优先从知识库中检索实验室安全条目，再结合模型生成结构化回答。这个问题不是问某一个细节操作，而是问高风险实验能不能开工。系统的回答重点会落在审批、风险评估、PPE、工程控制和应急准备这些开工前置条件上。”

可补一句：

“这套问答链路已经做过线上正式回归，最新 20 题是 `20/20`，不是只挑了几个好问题来演示。”

这一段要强调的点：

- 回答不是泛泛而谈，而是结构化输出。
- 回答背后有知识库支撑，不是纯模型自由发挥。
- 这类问题直接对应实验开始前的真实决策场景。

备用问题：

- `高风险实验可以一个人单独做吗？`
- `实验开始前要检查什么？`

### 1:00 - 1:35 开工前检查清单模块

操作：

1. 切到“开工前检查清单”。
2. 场景输入：`夜间进行高温回流和有机溶剂加热实验`
3. 故意保留一个关键项不勾选，例如“未完成监护/审批”。
4. 提交并展示被阻断的结果。

建议话术：

“聊天回答只能提供建议，但真正能降低事故率的是把建议变成阻断条件。这里我们把实验前必须确认的事项做成清单，包括 SOP、标签核对、PPE、通风/屏蔽、应急设施，以及高风险场景下的审批和双人监护。只要关键项不满足，系统就不允许开工。”

这一段要强调的点：

- 从“能问”升级到“能拦”。
- 关键项未通过时是阻断，而不是仅提醒。
- 这一步最能体现平台价值，不只是问答。

### 1:35 - 2:10 管理端看板模块

操作：

1. 切到管理端看板。
2. 依次指出以下指标：
  - 清单阻断率
  - 培训通过率
  - 低置信问题
  - 最近高风险场景

建议话术：

“除了面向个人使用，我们还做了管理端看板。老师和实验室管理员关心的不是单条回答，而是整体风险趋势。比如最近哪些场景容易被阻断、培训通过率如何、哪些问题属于低置信高频问题、哪里需要补知识条目。这样系统就具备持续迭代能力。”

可补一句：

“在验收层面，我们已经连续跑过 3 轮真实回归，核心正确率指标都是稳定通过的。”

这一段要强调的点：

- 平台可以支持管理，而不只是个人问答。
- 低置信问题会反向驱动知识库补强。
- 可以用数据支撑后续答辩中的“效果评估”。

### 2:10 - 2:45 事故复盘模块

操作：

1. 切到事故复盘。
2. 展示一条已有事故记录。
3. 指出字段：
  - 事件经过
  - 原因分类
  - 整改措施
  - 复发风险等级
  - 整改逾期提醒

建议话术：

“最后展示的是事故复盘模块。很多系统只做到预警，但实验室安全真正要闭环，还需要把事故或未遂事件记录下来，做原因分类、整改分配、逾期提醒和闭环验证。这样平台就不是只在事故前发挥作用，而是覆盖事前、事中、事后全过程。”

这一段要强调的点：

- 从预防延伸到复盘闭环。
- 能形成责任、时限、验证状态。
- 这一模块非常适合答辩时体现“管理价值”。

### 2:45 - 3:00 收尾

建议话术：

“所以我们当前交付的不是单点问答工具，而是一个实验室安全助手平台原型。它已经具备问答、阻断、看板、复盘四类能力。下一步我们会继续补高价值安全资料、扩充正式知识源，并在条件成熟后把云端模型切换到本地模型，进一步提升可控性和私有化落地能力。”

建议把最后一句口径固定为：

“截至目前，系统已经完成线上知识库导入、20题正式回归 `20/20`、连续 3 轮真实回归通过。当前剩余的重点工作是继续压缩延迟波动，而不是再去解决回答正确性。”

## 4. 现场提问时的备用回答

### 如果评委问：你们和普通 AI 问答有什么区别？

回答：

“普通 AI 偏重文本回答，我们的平台多了一层规则约束和流程化执行能力，尤其是开工阻断、事故复盘和管理看板，这是实验室场景真正需要的。”

### 如果评委问：为什么不直接做一个聊天机器人？

回答：

“因为实验室安全不是‘知道答案’就够了，更关键的是把安全要求落实到开工前检查、风险记录、整改闭环这些环节，所以我们做的是平台而不是单一问答框。”

### 如果评委问：现在最真实的短板是什么？

回答：

“当前最大的短板仍然是高质量、可追溯、可持续更新的正式知识源规模还不够大。我们已经把来源追溯、低置信回流和知识补强这条链路搭起来，后面重点就是持续扩容高价值官方资料。”

## 5. 现场演示注意事项

- 不要一开始讲太多技术细节，先把平台价值讲清楚。
- 问答只演示 1 个主问题，避免拖时间。
- 清单模块一定要演示一次“阻断”，这是最有辨识度的亮点。
- 看板模块不要逐个数字解释，只强调“能统计、能发现问题、能驱动补强”。
- 事故复盘部分突出“闭环”，不要只说“能记录”。
