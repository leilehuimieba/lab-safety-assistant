## 二、课题设计论证

### 2.1 研究背景与意义

实验室是高校教学与科研的重要场景，同时也是安全风险高度集中的场景。当前实验室安全管理普遍存在以下问题：

1. 安全知识分散在制度文件、SOP、MSDS、设备手册和教师经验中，检索效率低。
2. 学生与新进实验人员对危化品、设备操作和应急流程理解不一致，学习成本高。
3. 传统培训以集中授课和纸质文件为主，缺乏持续、即时、个性化的安全支持。
4. 事故隐患识别依赖人工经验，难以形成可追踪、可量化、可复盘的闭环。

本项目拟建设“实验室安全小助手”，提供“事前预防、事中提醒、事后复盘”一体化服务。其核心意义如下：

1. 管理意义：将分散安全知识整合为结构化知识库，提升日常管理效率与一致性。
2. 教学意义：面向学生和实验人员提供可追溯、可解释的安全问答与训练支持。
3. 技术意义：探索“规则引擎 + 大模型 + 检索增强”在高规范场景中的可信应用范式。
4. 推广意义：形成可复制的校内部署模板，为院系级实验室安全数字化提供样板。

### 2.2 国内外研究与应用现状（简述）

现有实验室安全智能化工作主要集中在三类路径：

1. 智能问答路径：以知识问答替代部分人工咨询，提升查询效率。
2. 感知监测路径：基于物联网传感器进行异常检测和预警。
3. 管理平台路径：将培训、台账、告警、考核整合为统一管理系统。

现有方案的共性短板主要是：

1. 模块割裂，问答系统与管理系统脱节。
2. 知识更新机制薄弱，制度变更后同步滞后。
3. 安全边界不清晰，对高风险问题缺少稳定的拒答与转人工策略。
4. 缺少针对高校实验场景的分角色服务能力。

本项目在吸收上述路径优势的基础上，重点补齐“知识更新、风险控制、角色分层、可验收指标”四个方面。

### 2.3 研究目标

#### 2.3.1 总体目标

构建一个可在高校实验室场景稳定运行的智能安全助手系统，面向学生、实验人员和管理员提供标准化问答、风险评估、应急引导和培训支持，形成可持续优化的安全服务闭环。

#### 2.3.2 具体目标

1. 知识库建设目标
- 建成不少于 3000 条结构化安全知识条目。
- 覆盖危化品、设备操作、实验室制度、PPE、防火防爆、废弃物处置、应急流程等核心主题。
- 为每条知识建立来源、版本、更新时间、适用范围等元数据。

2. 智能问答目标
- 支持自然语言问答与多轮追问。
- 回答中提供依据来源，提升可信度与可追溯性。
- 对低置信度问题触发“保守回答 + 人工复核提示”。

3. 风险评估目标
- 用户输入实验步骤、试剂和设备后，系统输出风险等级、关键危险点、必要防护、禁忌操作和建议流程。
- 对高风险内容触发规则约束，避免危险操作建议直接输出。

4. 管理与评估目标
- 建设管理端看板，支持高频问题统计、风险热力图、用户反馈分析。
- 建立离线评测与线上监控机制，支持迭代优化。

### 2.4 拟解决的关键问题

1. 问题一：专业术语和别名导致检索不稳定。
- 解决方案：构建术语词典和同义词映射，采用关键词检索与向量检索融合，增加重排模型。

2. 问题二：制度更新后知识滞后。
- 解决方案：建立“采集-清洗-审核-发布-回滚”流程，设置版本号和更新时间戳。

3. 问题三：高风险问题回答不够可控。
- 解决方案：引入规则引擎做前置约束与后置审查，设置拒答、转人工和应急模板响应策略。

4. 问题四：不同角色需求差异大。
- 解决方案：采用角色分层（学生/实验人员/管理员），提供差异化功能入口与权限控制。

### 2.5 项目创新点

1. 业务创新：从“单轮问答工具”升级为“问答 + 风险评估 + 应急引导 + 培训复盘”闭环系统。
2. 技术创新：采用“规则引擎 + RAG + 大模型”协同架构，提升安全场景可控性。
3. 工程创新：建立知识版本治理与回滚机制，保障知识服务的连续性与可审计性。
4. 应用创新：构建面向高校实验室的分角色交互体系，提高系统实际可用性。
5. 评估创新：将检索质量、回答质量、安全合规、系统性能统一纳入量化验收框架。

### 2.6 技术路线图

#### 2.6.1 整体架构

数据源层 -> 知识工程层 -> 安全推理层 -> 应用服务层 -> 评估迭代层

#### 2.6.2 各层说明

1. 数据源层
- 校内数据：制度文件、SOP、应急预案、设备手册。
- 外部数据：MSDS、国家标准与法规、典型事故案例。

2. 知识工程层
- 文档解析：对 PDF、Word、Excel 等进行统一解析。
- 清洗标准化：去重、字段映射、术语统一、风险标签标注。
- 知识索引：构建关键词索引（BM25）与向量索引并行体系。

3. 安全推理层
- 混合检索：关键词召回 + 向量召回 + 重排筛选。
- RAG 生成：基于检索片段生成回答并附依据来源。
- 规则护栏：高风险意图识别、危险指令拦截、应急模板优先响应。

4. 应用服务层
- 用户端：问答、风险评估、应急引导、学习训练。
- 管理端：知识管理、日志审计、统计报表。

5. 评估迭代层
- 采集指标：召回率、正确率、安全合规率、响应时延、满意度。
- 迭代优化：按周更新 Prompt、规则库和知识版本。

### 2.7 研究方法

1. 文献与规范研究法：梳理实验室安全管理规范、危化品管理标准和应急处置流程。
2. 需求分析法：通过访谈与问卷收集三类角色需求，形成功能优先级清单。
3. 原型迭代法：采用敏捷开发，分阶段实现 MVP、试点版、验收版。
4. 对比评测法：构建标准题集，对比“仅大模型问答”与“RAG+规则”效果差异。
5. 应用验证法：在试点实验室开展真实使用验证并收集反馈。

### 2.10 验收指标

#### 2.10.1 功能指标

1. 完成问答、风险评估、应急引导、知识管理四大核心模块。
2. 实现学生、实验人员、管理员三类角色分权限访问。

#### 2.10.2 数据指标

1. 知识条目数量不少于 3000 条。
2. 核心场景覆盖率不少于 90%。
3. 新增或修订知识从审核通过到上线不超过 72 小时。

#### 2.10.3 质量与安全指标

1. 检索召回率（Top-3）不低于 85%。
2. 专家抽检答案正确率不低于 88%。
3. 安全合规率不低于 95%。
4. 高风险问题拒答或转人工召回率不低于 95%。

#### 2.10.4 性能与可用性指标

1. 平均响应时间不超过 3 秒。
2. P95 响应时间不超过 5 秒。
3. 试点期系统可用性不低于 99%。

#### 2.10.5 应用效果指标

1. 完成不少于 2 个实验室场景试点。
2. 试点有效用户不少于 60 人。
3. 用户满意度不低于 85%。

## 三、完成课题的可行性分析

### 3.1 技术可行性

1. 技术栈成熟：RAG、向量检索、规则引擎、Web 服务均有成熟开源生态支撑。
2. 可分阶段交付：可先交付问答 MVP，再叠加风险评估与应急模块，降低一次性实施风险。
3. 可观测可迭代：通过日志与指标闭环持续优化，避免“做完即停”的一次性系统。

### 3.2 数据可行性

1. 数据来源明确：校内制度文件、SOP、应急预案与公开 MSDS 可形成稳定数据源。
2. 数据治理可执行：通过统一模板和审核机制保证数据质量与可追溯性。
3. 数据更新可持续：建立固定更新责任人和周期机制，保障时效。

### 3.3 团队与组织可行性

1. 研发分工明确：后端、前端、知识工程、测试评估分工实施。
2. 指导机制明确：指导教师定期评审，关键节点专家把关。
3. 管理机制明确：双周例会、月度里程碑、风险清单动态更新。

### 3.4 风险分析与对策

1. 风险：知识库质量不稳定。
- 对策：设置专家抽检与双人复核机制，建立回滚版本。

2. 风险：模型对高风险问题回答不当。
- 对策：规则前置拦截 + 结果后置审查 + 转人工策略。

3. 风险：试点参与度不足。
- 对策：与实验课程、岗前培训结合，明确试点激励与反馈机制。

4. 风险：项目后期时间紧张。
- 对策：采用最小可行路径，优先保障核心功能与验收指标达成。

## 五、伦理与合规说明

1. 本系统用于实验室安全教育与管理辅助，不替代专业人员现场判断。
2. 涉及人身伤害、火灾、泄漏等高危场景时，系统优先输出应急流程和求助建议。
3. 系统对日志数据仅用于改进服务质量，按校内数据管理制度执行权限控制和脱敏处理。
4. 对可能存在不确定性的回答，明确提示“仅供参考，请联系实验室管理员或指导教师确认”。
