## 四、踩过的坑（避免下次重复）

### 坑1：`{{context}}` 必须用 Dify 变量选择器插入
- **错误做法**：在系统提示词里直接手打 `{{context}}` 文本
- **正确做法**：在提示词编辑框中，光标定位后输入 `{`，从弹出的变量选择器中选择上下文变量（会显示为彩色标签）
- **症状**：LLM 节点上方出现橙色警告"要启用上下文功能，请在提示中填写上下文变量"，LLM 回答不使用知识库内容

### 坑2：结构化输出需关闭
- MiniMax-M2.5 不支持结构化输出（JSON Schema 模式）
- 开启状态下发布会报错"发布前确保所有问题均已解决"
- **解决**：LLM 节点设置 → 关闭"结构化输出"开关

### 坑3：Rerank 模型不配置不能留空
- 知识检索节点的召回设置里如果选了"Rerank 模型"模式，必须配置 Rerank 模型
- 当前没有 Rerank 模型 → **解决**：切换为"权重设置"模式（向量权重0.7 + 关键词0.3）

### 坑4：CSV 重上传后需等待重新索引
- 升级为 high_quality 后，旧文档要删掉重传
- 重传后要等 bge-m3 完成向量化（约需几分钟）
- 索引未完成时检索分数为0

### 坑5：Dify API 发布需要正确的 CSRF Token
- Cookie 里的 key 是 `csrf_token=`（不是 `_csrf_token=`）
- 浏览器控制台获取方式：
  ```javascript
  document.cookie.split(';').find(c=>c.trim().startsWith('csrf_token=')).split('=').slice(1).join('=')
  ```
- 发布接口：`POST /console/api/apps/{appId}/workflows/publish`

## 六、我的思考

### 关于项目定位
这个项目是一个 **RAG 原型**，不是完整产品。核心价值在于展示"私有知识库 + LLM = 专域问答助手"的技术路线，适合大创/课程答辩场景。

### 关于知识库质量
81条是够用的演示规模，但覆盖面仍有限。知识条目的质量比数量更重要——每条都应有清晰的 `question`、`answer`、`steps`、`ppe`、`forbidden` 字段，这样检索出来的内容对 LLM 最友好。

### 关于模型选择
MiniMax-M2.5 是合理的选择（国内可访问、支持长上下文、有推理能力）。如果未来想升级，可以考虑：
- 本地部署：Ollama + Qwen2.5（完全离线，无 API 费用）
- 在线：DeepSeek-V3（性价比高，中文表现好）

### 关于 RAG 流程的关键洞察
整个流程里最容易出错的地方不是模型，而是**上下文变量的连接**。Dify 的 `{{context}}` 是纯文本占位符，必须通过 UI 变量选择器插入才能生效。这是一个反直觉的设计，容易踩坑。

### 关于下一步方向
如果要把这个项目真正做成"智能体"（而不只是问答机器人），可以考虑：
1. 在 Chatflow 里加"条件判断"节点（根据问题类型走不同分支）
2. 加入工具调用（如查询实验室排班表、提交报修工单）
3. 接入多知识库（不同实验室各自的规范文档）
