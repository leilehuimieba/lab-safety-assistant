# 实验室安全小助手数据源建设方案

## 文档目的

本方案用于说明“实验室安全小助手”下一阶段应如何建设正式数据源，使系统从当前的 MVP 样例知识库，逐步过渡到来源清晰、结构稳定、可持续更新的实验室安全知识体系。

## 为什么要补正式数据源

当前仓库中的 `knowledge_base_curated.csv` 已经可以支撑原型展示和初步问答，但它的定位更接近“核心场景样例集”，并不等同于完整正式知识体系。

如果项目后续要用于以下场景，就必须引入正式数据源：

- 项目结题或答辩中的可信性说明
- 校内实验室安全培训辅助
- 更高覆盖率的问答系统
- 更强的回答可追溯性
- 后续本地模型与评测对比实验

## 建设目标

数据源建设阶段的目标不是单纯堆积文件，而是建立一套可管理的知识来源体系，确保：

- 来源明确
- 权限可控
- 内容可清洗
- 字段可结构化
- 更新可追踪
- 回答可追溯

## 数据源优先级划分

建议按优先级分三层推进。

## 第一层：必须优先建设的数据源

这些数据源最适合直接提升系统可信度和答辩说服力。

### 1. 校内实验室安全管理制度

建议收集内容：

- 实验室安全管理办法
- 学生实验守则
- 危险化学品管理制度
- 实验室准入和培训制度
- 应急管理制度

用途：

- 回答校内合规类问题
- 强化系统“制度依据”属性
- 作为项目落地到本校场景的核心基础

### 2. 典型实验室 SOP

建议收集内容：

- 通风柜使用 SOP
- 危化品领用与储存 SOP
- 废液分类与收集 SOP
- 电气设备上电/断电 SOP
- 生物样本处理 SOP

用途：

- 回答操作规范类问题
- 将制度条文转化为可执行步骤
- 提升回答的“动作性”和“现场指导性”

### 3. 常用危化品 MSDS

建议优先覆盖：

- 乙醇
- 丙酮
- 盐酸
- 硫酸
- 硝酸
- 氢氧化钠
- 漂白剂相关试剂

用途：

- 回答储存、使用、防护、泄漏和急救问题
- 为危险化学品问答提供专业依据

## 第二层：强烈建议建设的数据源

### 4. 实验室消防与电气应急预案

建议收集内容：

- 火灾应急预案
- 触电事故应急预案
- 危险化学品泄漏应急预案
- 气瓶和气体泄漏应急处置流程

用途：

- 强化应急场景回答
- 让系统在高风险问题上更有步骤感和优先级逻辑

### 5. 实验室安全培训课件和考试题

建议收集内容：

- 新生实验室安全培训 PPT
- 安全教育题库
- 常见违规操作案例

用途：

- 扩展面向学生的解释型问答
- 辅助构建“培训版”问答场景

### 6. 设备说明书与使用规范

建议优先设备：

- 通风柜
- 高压电源
- 离心机
- 加热设备
- 气瓶及减压阀

用途：

- 回答设备使用前检查、操作规范和风险提示问题

## 第三层：可选扩展数据源

### 7. 国家标准、行业标准与公开安全规范

适用范围：

- 补充通用安全知识
- 作为规则设计参考
- 在校内制度尚不完整时提供补充

注意事项：

- 优先引用权威公开标准
- 避免直接把不相关工业标准大批量倒入知识库
- 必须做主题筛选，否则容易带来检索噪声

### 8. 实验事故案例库

用途：

- 用于警示教育
- 用于丰富“为什么危险”的解释能力

注意事项：

- 不宜直接作为主知识源
- 更适合作为教学补充材料

## 数据源建设原则

### 原则 1：先少量高质量，再大批量扩充

不要一开始把大量来源不明、主题不聚焦的 PDF 一次性导入知识库。  
优先做少量高质量、强相关、可追溯的数据源。

### 原则 2：每条知识都要有来源字段

建议后续每一条知识都补齐以下字段：

- `source_type`
- `source_title`
- `source_org`
- `source_version`
- `source_date`
- `source_url`
- `last_updated`
- `reviewer`
- `status`

### 原则 3：优先结构化，而不是只保留原文

实验室安全问答更需要“问题 -> 结论 -> 步骤 -> 防护 -> 禁止事项 -> 应急”的结构化表达，而不是长篇原文堆积。

### 原则 4：校内制度优先于泛化资料

如果校内制度与外部公开资料有差异，系统回答应优先遵循校内制度。

## 推荐的数据字段结构

建议继续沿用当前知识库样例的结构思路。每条知识至少应包含：

- `id`
- `title`
- `category`
- `subcategory`
- `lab_type`
- `risk_level`
- `hazard_types`
- `scenario`
- `question`

- `forbidden`

## 数据处理流程建议

建议采用“五步法”进行数据建设。

### 第一步：收集

按来源类型建立原始资料目录：

- 校内制度
- SOP
- MSDS
- 应急预案
- 培训材料
- 设备说明书

### 第二步：筛选

筛选标准：

- 是否与实验室安全直接相关
- 是否适合转化为问答知识
- 是否有明确出处
- 是否存在时效性问题

### 第三步：清洗

清洗重点：

- 去掉封面、目录、页眉页脚、无关附录
- 提取关键条款和步骤
- 删除明显与实验室安全主题无关的内容

### 第四步：结构化

将原始资料整理成统一条目格式，优先抽取：

- 适用场景
- 核心结论
- 操作步骤
- 个人防护要求
- 禁止事项
- 应急建议

### 第五步：复核

建议每批条目都做一次人工复核，至少检查：

- 事实是否准确
- 语言是否清晰
- 是否有来源字段
- 是否有安全边界

## 数据源目录建议

如果后续要在仓库中逐步沉淀来源材料，建议采用以下目录逻辑：

```text
data_sources/
├─ institutional/
├─ sop/
├─ msds/
├─ emergency/
├─ training/
└─ equipment/
```

注意：

- 原始文件不建议直接全部公开提交
- 若涉及校内内部材料或版权问题，应只提交清洗后的结构化结果和来源元数据

## 与规则库的配合方式

数据源解决的是“知识依据”问题，规则库解决的是“回答约束”问题。

两者建议分工如下：

- 数据源：提供事实、制度、步骤、要求
- 规则库：决定何时拒答、何时警示、何时优先输出应急步骤

例如：

- “盐酸和漂白水能混吗？”  
  数据源负责提供混合禁忌事实  
  规则库负责保证回答方式以禁止和警示为主

## 与评测集的配合方式

每增加一批正式数据源，建议同步做两件事：

- 扩展 `eval_set_v1.csv` 中相关问题
- 检查 `eval_criteria.md` 是否需要补充新场景指标

这样可以形成“数据源扩充 -> 系统测试 -> 指标验证 -> 调整优化”的闭环。

## 分阶段建设建议

### 第一阶段：校内制度与基础 SOP

目标：

- 补齐校内实验室安全制度
- 补齐通风柜、废液、危化品和培训相关 SOP

完成标志：

- 新增 30 至 50 条高质量知识条目
- 每条都带来源字段

### 第二阶段：MSDS 与应急预案

目标：

- 补齐常用危化品 MSDS
- 补齐火灾、触电、泄漏、灼伤等预案内容

完成标志：

- 高风险场景回答显著更具体
- 应急类问题命中率和步骤完整性提升

### 第三阶段：设备与扩展场景

目标：

- 引入设备说明书和实验事故案例
- 扩展生物、电气、消防和管理类场景覆盖

完成标志：

- 场景覆盖面明显扩大
- 系统适合做更完整的答辩演示

## 阶段性验收建议

可以把数据源建设成效用以下指标衡量：

- 有正式来源的知识条目数量
- 高风险场景覆盖数量
- 每条知识的来源字段完整率
- 评测集中命中关键要点的比例
- 应急类问题回答完整率

## 当前最推荐的下一步动作

如果现在只能先做一件事，建议优先做：

- 收集本校实验室安全制度
- 收集通风柜、危化品和废液处理 SOP
- 收集 5 到 10 种高频危化品 MSDS

原因是这三类资料最能直接提升：

- 答辩可信度
- 问答正确率
- 场景覆盖度
