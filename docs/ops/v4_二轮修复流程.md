# v4 二轮修复流程（自动聚类 + 自动改写 + 二次复检）

## 1. 目标

针对 `web_seed_v4` 的低分条目，执行以下自动化链路：

1. 首轮 AI 审核（baseline）
2. 自动生成低分原因聚类报告
3. 自动批量改写 `needs_fix/reject` 条目
4. 第二轮 AI 审核
5. 第二轮 AI 复检（recheck）
6. 输出前后对比报告

## 2. 一键执行命令

```powershell
python scripts\run_v4_repair_round.py `
  --repo-root . `
  --input-csv artifacts\web_seed_v4_prefetch\knowledge_base_web.csv `
  --output-root artifacts\v4_repair_round `
  --openai-base-url "https://api.tabcode.cc/openai" `
  --openai-api-key "你的apikey" `
  --openai-model "gpt-5.2-codex" `
  --openai-api "openai-responses" `
  --openai-insecure-tls `
  --strict-high-risk
```

## 3. 输出目录结构

每次运行会在 `artifacts/v4_repair_round/run_YYYYMMDD_HHMMSS/` 下生成：

- `baseline_audit/`：首轮审核结果
- `baseline_cluster/failure_clusters.{csv,md}`：首轮低分聚类
- `rewrite/knowledge_base_rewritten.csv`：自动改写后的候选库
- `rewrite/rewrite_log.csv`：改写日志
- `second_audit/`：第二轮审核结果
- `second_cluster/failure_clusters.{csv,md}`：第二轮低分聚类
- `second_recheck/`：第二轮复检结果
- `repair_round_report.{json,md}`：整轮对比报告

## 4. 建议关注的三个指标

1. `baseline pass_rate -> second_audit pass_rate` 是否明显提升  
2. `post_rule_hits` 是否从高位下降到 0（说明字段完整性问题已修复）  
3. 第二轮仍未通过条目的共性（通常是来源可核验性、语言标签一致性、引用精确度）

## 5. 当前已知限制

1. 原始来源本身失效（404）的条目，无法仅靠改写达到高可信通过。  
2. 若 `language` 元数据与内容语言不一致，会显著拉低评分。  
3. 单一来源难以支撑非常具体的操作阈值，建议改为“按本单位制度和法规执行”的保守表述，或补充二级来源。
