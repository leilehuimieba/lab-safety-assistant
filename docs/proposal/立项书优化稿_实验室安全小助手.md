## 一、先做删减与校正（建议你在原稿中先删/改）

1. 删除或降级“绝对化表述”
- 建议删除：`首创`、`100%拦截`、`99%+无条件准确率`、`彻底杜绝`。
- 建议改为：`在试点数据集上达到`、`目标值`、`阶段性验证`。

2. 删除“无证据来源的外部数据”
- 如“某高校3分钟诊断”“某框架31.6%提升”等，若无可追溯文献或官方来源，建议改成“已有研究显示/相关研究表明”。

3. 合并重复章节，避免“研究意义”与“课题研究意义”重复
- 原稿中第1部分和第4部分有明显重复，建议保留一处完整论述，另一处改成“拟解决问题与创新点”。

4. 把“技术路线”与“国内外现状”分开写
- “国内外研究现状”写别人做了什么。
- “技术路线图”只写你们准备怎么做、怎么验证。

5. 调整指标到“可验收口径”
- 不建议写“专业问题准确率>99%”这类宽泛指标。

## 二、六大核心章节（详细替换稿）

## 1. 研究目标（建议替换“3.3 拟完成目标”）

### 1.1 总体目标
构建面向高校实验室的“实验室安全小助手”系统，形成“事前预防-事中提醒-事后复盘”的安全服务闭环。系统可为学生、实验人员、实验室管理员提供规范化问答、风险评估、应急处置指导和培训支持，提升实验室安全治理的数字化与智能化水平。

### 1.2 分目标
1. 建设高质量安全知识库
- 汇聚并结构化整理不少于3000条安全知识条目，覆盖危化品、设备操作、个体防护、废弃物处理、应急处置、实验室管理制度等核心场景。
- 建立知识条目元数据体系（来源、版本、更新时间、适用实验类型、风险等级、责任部门）。

2. 实现高可用问答与风险识别能力
- 在常规问答外，新增“实验步骤风险评估”功能，输出风险等级、关键危险点、必要PPE、禁止事项与替代建议。
- 实现“高风险问题分级响应”，对危险指令类问题进行拒答、转人工或仅输出规范流程。

3. 建立可持续更新机制
- 形成“采集-清洗-审核-发布-回滚”的知识更新流程。
- 支持版本对比与更新记录，确保制度更新可追溯。

4. 构建多角色服务能力
- 学生端：安全问答、术语解释、训练题与错题反馈。
- 实验人员端：风险评估、操作前检查清单、应急卡片。
- 管理员端：知识管理、问题热力图、风险事件统计报表。

5. 完成试点部署与效果评估
- 在至少2个实验室场景开展试点运行，输出系统评估报告和优化建议。

### 1.3 阶段目标（9个月）
- 第1阶段（M1-M2）：完成需求调研、指标体系设计、首批知识入库（≥1200条）。
- 第2阶段（M3-M4）：完成RAG问答原型与管理后台初版；上线基础安全过滤。
- 第3阶段（M5-M6）：上线风险评估、应急处置引导、多角色权限。
- 第4阶段（M7-M8）：完成试点运行与评测闭环；根据数据迭代优化。
- 第5阶段（M9）：形成验收材料、技术文档、成果汇编与推广方案。

## 2. 创新点（建议替换“4.3 主要创新之处”）

1. 从“单轮问答”升级为“安全闭环服务”
- 不是只做聊天机器人，而是打通“问答-风险评估-应急指导-培训复盘”四个环节，直接对应实验室安全管理流程。

2. “规则引擎 + 大模型”双引擎协同
- 对高风险场景优先采用规则约束（禁混试剂、危险操作阈值、应急流程约束），大模型负责解释与交互，提高可靠性与可控性。

3. 面向安全场景的检索增强架构
- 采用“关键词检索 + 语义检索 + 重排模型”混合检索流程，提升专业术语、缩写、别名场景下的检索稳定性。

4. 知识库版本治理与追溯机制
- 每条知识绑定来源和版本号，支持更新日志和回滚；回答可显示“依据来源”，增强可信度和可审计性。

5. 多角色分层服务
- 针对学生、实验人员、管理员提供差异化功能视图，实现“同一底座、分层能力输出”。

6. 可扩展多模态能力（进阶）
- 在资源允许时接入图像识别模块，支持危险标识识别、试剂标签核验、PPE佩戴检查。

## 3. 技术路线图（建议新增独立小节）

### 3.1 总体路线（文本图）

`数据源层 -> 知识工程层 -> 安全推理层 -> 应用服务层 -> 评估迭代层`

1. 数据源层
- 校内资料：实验室制度、SOP、应急预案、设备操作手册。
- 外部资料：MSDS、国家/行业规范、典型事故案例。

2. 知识工程层
- 文档解析：PDF/Word/表格统一解析。
- 数据清洗：去重、术语标准化、字段对齐。
- 结构化建模：构建知识条目模板（适用场景、风险等级、处置步骤、禁忌事项、引用来源）。
- 向量化与索引：建立向量索引与关键词索引并行的混合检索索引。

3. 安全推理层
- 检索策略：BM25关键词检索 + 向量检索 + Cross-Encoder重排。
- 生成策略：RAG增强生成，回答附来源引用。
- 安全策略：风险分级、敏感意图识别、危险指令拦截、人工复核接口。

4. 应用服务层
- 核心功能：安全问答、风险评估、应急引导、培训测评、管理看板。
- 接口服务：Web端/移动端统一API。

5. 评估迭代层
- 日志采集：问题类型、命中来源、响应时间、用户反馈。
- 指标评估：检索效果、回答质量、安全合规、用户满意度。
- 持续优化：按周迭代Prompt、检索参数、规则库与知识库版本。

### 3.2 关键技术清单（先进但可落地）
- RAG：向量检索+关键词检索混合检索。
- 重排模型：提升检索命中质量，减少“检索到但不相关”。
- 安全护栏：规则引擎、拒答策略、风险分级模板。
- 评测框架：离线题集 + 专家打分 + 线上日志回归测试。
- 权限控制：分角色访问控制（RBAC）和审计日志。

## 6. 验收指标（建议新增独立小节，强调“可测”）

### 6.1 功能指标
1. 完成问答、风险评估、应急引导、知识管理四个核心模块。
2. 支持三类角色（学生、实验人员、管理员）分权限访问。

### 6.2 数据与知识指标
1. 知识库条目数≥3000条。
2. 核心场景覆盖率≥90%（以项目定义的场景清单为准）。
3. 新知识更新时间≤72小时（从审核通过到上线）。

### 6.3 模型与效果指标
1. 检索召回率（Top-3）≥85%。
2. 专家评测下答案正确率≥88%。
3. 安全合规率≥95%（回答是否符合安全规范）。
4. 高风险问题拒答/转人工召回率≥95%。

### 6.4 性能与可用性指标
1. 平均响应时间≤3秒，P95响应时间≤5秒。
2. 试点期系统可用性≥99%。

### 6.5 应用与推广指标
1. 完成至少2个实验室场景试点。
2. 试点有效用户数≥60人。
3. 用户满意度≥85%。
4. 完成1份《试点应用效果报告》并通过导师组评审。

### 6.6 验收方式
- 离线测试：标准题集自动评分+专家抽检。
- 在线测试：试点日志统计（响应时延、调用成功率、拦截效果）。
- 材料验收：系统演示、部署文档、测试报告、用户反馈记录。

## 三、可选补充（答辩加分）

1. 伦理与合规声明
- 明确系统边界：系统不替代专业人员判断；涉及人身风险时给出“立即联系实验室管理人员/急救电话”提示。

2. 风险预案
- 当模型置信度低或检索结果不足时，触发“保守回答+人工复核”机制。

3. 推广路径
- 先在单院系试点，再扩展到全校实验室，逐步形成校级安全知识服务平台。
