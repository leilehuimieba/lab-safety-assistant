# 角色1：信息收集员执行手册 v2

## 1. 你的职责（只做这三件事）
1. 收集高价值、可追溯的实验室安全资料链接。
2. 按字段规范维护 `data_sources/web_seed_urls_v2_candidates.csv`。
3. 优先补齐预抓取失败/被拦截来源，提交 PR 给清洗同学接力。

你不负责数据清洗、不负责模型调参。

## 2. 你先看哪里
仓库主页：
- `https://github.com/leilehuimieba/lab-safety-assistant`

你每次开始工作先打开这两个文件：
- `https://github.com/leilehuimieba/lab-safety-assistant/blob/main/data_sources/web_seed_urls_v2_candidates.csv`
- `https://github.com/leilehuimieba/lab-safety-assistant/blob/main/data_sources/web_seed_urls_v2_prefetch_status.csv`

其中：
- `web_seed_urls_v2_candidates.csv`：你负责新增/更新来源主表。
- `web_seed_urls_v2_prefetch_status.csv`：告诉你哪些来源被拦截、哪些可以直接入库。

## 3. 本周优先级（按这个顺序做）
1. 先处理 `web_seed_urls_v2_prefetch_status.csv` 里 `status != ok` 的来源。
2. 再新增 `P0` 来源（法规、国家标准、高校制度）。
3. 最后补 `P1` 来源（高校手册、供应商 SDS、国际指南）。

优先级定义：
- `P0`：国家法规、教育部/应急管理部/生态环境部等权威来源，或核心高校制度。
- `P1`：高校公开手册、国际机构指南、供应商 SDS 门户。
- `P2`：案例、延展资料（当前批次可暂缓）。

## 4. 你要收集的来源类型
必收类型（至少覆盖其中三类）：
- 法规标准类：教育部、应急管理部、生态环境部、政府网法规页面。
- 高校制度类：实验室准入、危化品管理、废弃物处置、应急预案。
- 化学安全类：SDS/MSDS 检索入口与关键物质安全卡。
- 生物安全类：生物实验室分级、病原微生物管理、个人防护要求。

拒收类型：
- 无来源转载、无法追溯原文。
- 广告/营销页面。
- 与实验室安全弱相关内容。

## 5. CSV 填写规则（必须）
目标文件：
- `data_sources/web_seed_urls_v2_candidates.csv`

必须填写字段：
1. `source_id`：唯一编号，格式 `WEBV2-XXX`。
2. `title`：页面标题或法规名称。
3. `source_org`：发布机构。
4. `category`：如 `通用/制度/标准/SDS/化学/生物`。
5. `subcategory`：如 `国家规范/安全手册/安全管理`。
6. `lab_type`：如 `通用/化学/生物/电气/辐射`。
7. `risk_level`：`1-5`。
8. `hazard_types`：可多值分号分隔，如 `危化品;化学`。
9. `url`：原始公开链接。
10. `tags`：已有标签，分号分隔。
11. `suggested_tags`：建议标签，分号分隔。
12. `priority`：`P0/P1/P2`。
13. `language`：如 `zh-CN/en/en-US`。
14. `question_hint`：可用于问答的提示问题。

填写前先对照字典：
- `https://github.com/leilehuimieba/lab-safety-assistant/blob/main/data_sources/field_dictionary.json`

## 6. 你每天最小交付
1. 新增或修复至少 5 条来源记录。
2. 每条记录都包含 `priority` 和 `question_hint`。
3. 至少处理 1 条 `status != ok` 的来源（人工访问确认后补信息）。

## 7. GitHub 提交流程
分支命名：
- `feat/collect-<name>-<yyyymmdd>`

提交信息示例：
- `data: update web seed v2 sources and blocked entries (2026-03-18)`

PR 标题模板：
- `[数据收集-v2] <日期> <姓名> 新增/修复 <N> 条来源`

PR 描述必须包含：
1. 本次新增条数与修复条数。
2. `P0/P1/P2` 分布。
3. 是否处理了 `status != ok` 条目，处理了哪些 `source_id`。
4. 是否包含需人工登录访问来源（如有需说明）。

## 8. 交接给清洗同学前自检
1. URL 可打开，且对应标题与机构无误。
2. `risk_level`、`category`、`lab_type` 合理。
3. `question_hint` 不是空话，能直接用于问答测试。
4. 没有重复 `source_id`，没有空 URL。

## 9. 常见错误（避免）
1. 只贴链接不填元数据。
2. `hazard_types` 写成逗号，正确应为分号分隔。
3. 新增来源不标优先级，导致清洗排期混乱。
4. 收集大量 P2，忽略 P0 的法规标准类。

## 10. 抓取失败时的人工补录路径
当某条来源 `error/blocked`，但你能手工获取 PDF 附件时，走人工补录：

1. 文件放到 `manual_sources/inbox/`
2. 在 `data_sources/manual_document_manifest.csv` 新增 1 行元数据
3. 在 `tags` 里建议加 `替代WEBV3-xxx`，保留和原失败条目的关联

详细说明见：
- `docs/ops/manual_pdf_intake_workflow.md`
- `docs/ops/v3_采集补链与人工补录说明.md`

## 11. 你交付后谁接手
你的 PR 合并后，由“数据清洗同学”按以下文件继续执行：
- `data_sources/web_seed_urls_v2_candidates.csv`
- `data_sources/web_seed_urls_v2_prefetch_status.csv`
